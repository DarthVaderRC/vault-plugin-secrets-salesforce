# Salesforce secrets engine (API)

This is the HTTP API reference for the Salesforce secrets engine. The engine is
mounted at `salesforce/` by default; adjust the path in the examples to match
your mount. For an overview, see the
[Salesforce secrets engine documentation](/vault/docs/secrets/salesforce).

## Create or update a configuration

Creates or updates the connection configuration for one Salesforce org and app.
Secret fields are write-only and never returned by a read.

| Method | Path |
|--------|------|
| `POST` | `/salesforce/config/:name` |

### Parameters

- `name` `(string: <required>)` – Name of the configuration. Provided as part of
  the request path.
- `login_url` `(string: <required>)` – Base login host, for example
  `https://login.salesforce.com`, `https://test.salesforce.com`, or
  `https://<MyDomain>.my.salesforce.com`.
- `client_id` `(string: <required>)` – Connected App / External Client App
  Consumer Key (OAuth `client_id`).
- `client_secret` `(string: "")` – Consumer Secret. Required for
  `client_credentials` roles. Write-only; never returned.
- `private_key` `(string: "")` – PEM RSA private key for JWT Bearer signing.
  Required for `jwt_bearer` roles. Write-only; never returned.
- `token_url` `(string: "")` – Override for the full token endpoint. Defaults to
  `<login_url>/services/oauth2/token`.
- `ca_cert` `(string: "")` – PEM CA bundle used to validate the Salesforce TLS
  endpoint.
- `tls_skip_verify` `(bool: false)` – Disable TLS verification. Sandbox and
  testing only. Changing this on an existing config requires re-supplying the
  secret material (see the note below).
- `allow_non_salesforce_host` `(bool: false)` – Allow the token endpoint to target
  a non-Salesforce host (and loopback). When `false`, the host must end in
  `.salesforce.com`, `.force.com`, or `.salesforce.mil`. Loopback addresses
  require this opt-in. Changing this on an existing config requires re-supplying
  the secret material (see the note below).

<Warning>

Secret fields (`client_secret`, `private_key`) are write-only and preserved
across updates. An update that repoints the token endpoint (`token_url` or
`login_url`) or relaxes a transport guard (`allow_non_salesforce_host` or
`tls_skip_verify`) **must re-supply** `client_secret` and/or `private_key` in the
same request. This prevents redirecting the stored secret to an unintended host.

</Warning>

### Sample payload

```json
{
  "login_url": "https://acme.my.salesforce.com",
  "client_id": "3MVG9...",
  "client_secret": "9F8E7D..."
}
```

### Sample request

```shell-session
$ curl \
    --header "X-Vault-Token: ..." \
    --request POST \
    --data @payload.json \
    http://127.0.0.1:8200/v1/salesforce/config/prod
```

## Read a configuration

Returns the configuration with secret fields redacted. `client_secret` and
`private_key` show `<redacted>` when set and `""` when unset.

| Method | Path |
|--------|------|
| `GET`  | `/salesforce/config/:name` |

### Sample request

```shell-session
$ curl \
    --header "X-Vault-Token: ..." \
    http://127.0.0.1:8200/v1/salesforce/config/prod
```

### Sample response

```json
{
  "data": {
    "login_url": "https://acme.my.salesforce.com",
    "token_url": "https://acme.my.salesforce.com/services/oauth2/token",
    "client_id": "3MVG9...",
    "client_secret": "<redacted>",
    "private_key": "",
    "ca_cert": "",
    "tls_skip_verify": false,
    "allow_non_salesforce_host": false
  }
}
```

## Create or update a role

Creates or updates a role that issues tokens via a grant flow and identity. The
engine validates flow invariants on write: a `jwt_bearer` role requires
`username` and a config `private_key`; a `client_credentials` role requires a
config `client_secret`.

| Method | Path |
|--------|------|
| `POST` | `/salesforce/roles/:name` |

### Parameters

- `name` `(string: <required>)` – Name of the role. Provided as part of the
  request path.
- `config` `(string: <required>)` – The `config/<name>` this role uses.
  Immutable after the role is created.
- `grant_type` `(string: <required>)` – `jwt_bearer` or `client_credentials`.
- `username` `(string: "")` – Salesforce username for the JWT `sub` claim.
  Required for `jwt_bearer`. Immutable after the role is created.
- `scopes` `(list: [])` – OAuth scopes. The Client Credentials flow ignores
  request scopes; set them on the Connected App.
- `token_ttl` `(duration: "15m")` – Assumed access-token lifetime; drives cache
  expiry.
- `ttl` `(duration: "")` – Default Vault lease TTL for issued tokens.
- `max_ttl` `(duration: "")` – Maximum Vault lease TTL for issued tokens.
- `renew_skew` `(duration: "60s")` – Re-mint this long before token expiry. Must
  be `>= 0` and strictly less than `token_ttl`.
- `jwt_expiry` `(duration: "3m")` – JWT assertion `exp` window. Maximum `5m`.
- `audience` `(string: "")` – Override the JWT `aud`. Defaults to the config
  `login_url`.
- `revoke_tokens` `(bool: false)` – When `true`, call the Salesforce `/revoke`
  endpoint on lease revoke and on rotate. Because the token is shared across all
  leases of the role, this invalidates it for every current holder.
- `use_introspection` `(bool: false)` – Reserved; not yet implemented (no
  effect). Setting it returns a warning.

<Note>

`username` and `config` are immutable after a role is created. To change the
Salesforce identity or the bound config, delete and recreate the role.

</Note>

### Sample payload

```json
{
  "config": "prod",
  "grant_type": "client_credentials",
  "token_ttl": "15m",
  "ttl": "10m",
  "max_ttl": "1h"
}
```

### Sample request

```shell-session
$ curl \
    --header "X-Vault-Token: ..." \
    --request POST \
    --data @payload.json \
    http://127.0.0.1:8200/v1/salesforce/roles/sales-sync
```

## Issue a token

Returns a leased Salesforce access token for the role. The token is cached per
role and re-minted near expiry. The path `/salesforce/token/:name` is an
identical alias.

| Method | Path |
|--------|------|
| `GET`  | `/salesforce/creds/:name` |
| `GET`  | `/salesforce/token/:name` |

<Warning>

`token/:name` is a functional alias of `creds/:name`. When writing ACL policies,
apply the **same** rule to both paths: a `deny` or omission on `creds/*` is
bypassable through `token/*`.

</Warning>

### Sample request

```shell-session
$ curl \
    --header "X-Vault-Token: ..." \
    http://127.0.0.1:8200/v1/salesforce/creds/sales-sync
```

### Sample response

```json
{
  "lease_id": "salesforce/creds/sales-sync/AbCdEf...",
  "lease_duration": 600,
  "renewable": true,
  "data": {
    "access_token": "00D5g000004...",
    "instance_url": "https://acme.my.salesforce.com",
    "token_type": "Bearer",
    "expires_at": "2026-06-25T06:15:00Z",
    "grant_type": "client_credentials",
    "cached": false
  }
}
```

The `cached` field reports whether the token was served from the cache. Use the
`access_token` against `instance_url` for subsequent Salesforce API calls.

## Rotate a role token

Discards the role's cached token and mints a new one immediately. Subsequent
reads of `creds/:name` serve the new token. Previously issued tokens for the role
remain valid at Salesforce until they expire, unless the role sets
`revoke_tokens=true`, in which case the engine calls Salesforce's `/revoke` for
the outgoing token (invalidating it for all current holders).

| Method | Path |
|--------|------|
| `POST` | `/salesforce/roles/:name/rotate` |

### Sample request

```shell-session
$ curl \
    --header "X-Vault-Token: ..." \
    --request POST \
    http://127.0.0.1:8200/v1/salesforce/roles/sales-sync/rotate
```

### Sample response

```json
{
  "data": {
    "rotated": true,
    "role": "sales-sync",
    "expires_at": "2026-06-25T06:30:00Z"
  }
}
```
