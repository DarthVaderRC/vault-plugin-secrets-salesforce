# How the Salesforce secrets engine works

The Salesforce secrets engine turns a stored Salesforce credential into
short-lived OAuth access tokens. This page explains the data model, the two
grant flows, and how the engine caches tokens and maps them to Vault leases.

## Data model

The engine separates long-lived connection details from per-consumer token
policy, following the same `config` / `roles` pattern as Vault's database and
cloud secrets engines.

### Configuration

A *configuration* (`config/<name>`) describes one Salesforce org and Connected
App. It stores the login host, the OAuth `client_id`, and the secret material
for the grant flows you intend to use.

Secret fields — `client_secret` and `private_key` — are **write-only**. The
engine stores them in the Vault barrier and never returns them. A read of a
configuration shows `<redacted>` in place of any secret that is set.

### Role

A *role* (`roles/<name>`) binds a configuration to a single grant flow and an
identity, then layers Vault TTL policy on top. The role's `grant_type` selects
the flow:

- `client_credentials` requires the bound config to have a `client_secret`.
- `jwt_bearer` requires a `username` (the run-as identity) and the bound config
  to have a `private_key`.

The engine validates these invariants when you write the role, so a misconfigured
role fails fast rather than at token-issue time.

### Credentials

A *credential* read (`creds/<name>`, or the alias `token/<name>`) returns an
access token for the role. The response includes the `access_token`, the
`instance_url` to call, the `token_type`, an `expires_at` timestamp, the
`grant_type`, and a `cached` boolean that reports whether the token came from the
cache.

## Grant flows

Both supported flows are server-to-server grants that issue an access token
without a refresh token and without interactive user approval.

### Client Credentials

The engine posts `grant_type=client_credentials` with the config's `client_id`
and `client_secret` to the token endpoint. The token acts as the Connected App's
"Run As" user.

<Note>

Salesforce's Client Credentials flow does **not** accept a `scope` request
parameter — sending one fails with `invalid_request: scope parameter not
supported`. The granted scopes come from the Connected App's OAuth configuration,
so the engine does not forward a role's `scopes` for this flow.

</Note>

### JWT Bearer

The engine builds a JWT assertion, signs it with RS256 using the config's
`private_key`, and exchanges it at the token endpoint for an access token. The
token acts as the Salesforce user named in the role's `username`.

The assertion claims follow the Salesforce JWT Bearer flow:

| Claim | Value |
|---|---|
| `iss` | The config `client_id` (Consumer Key). |
| `sub` | The role `username` (run-as identity). |
| `aud` | The config `login_url`, or the role `audience` override. Many orgs use `https://login.salesforce.com`. |
| `exp` | `now + jwt_expiry`. Defaults to 3 minutes; Salesforce rejects assertions dated too far in the future, so the engine caps `jwt_expiry` at 5 minutes. |

## Token caching

The engine caches **one token per role** in the Vault barrier at `cache/<role>`.
A read serves the cached token when it is still *fresh* and otherwise mints a new
one.

A token is fresh when:

```text
now < expires_at - renew_skew
```

`renew_skew` (default 60 seconds) re-mints the token slightly before it actually
expires, so consumers never receive a token that is about to lapse. Because a
burst of reads shares one cached token, the engine does not churn tokens or
exhaust Salesforce's per-user token allocation.

### Expiry derivation

Salesforce usually omits `expires_in` from the token response. The engine derives
the cache expiry as follows:

1. If the response includes `expires_in`, use `now + expires_in`.
2. Otherwise, use `now + token_ttl` (the role's assumed lifetime, default
   15 minutes).

Set `token_ttl` to match the session timeout configured in your Salesforce org so
the cache does not consider a token fresh after Salesforce has expired it.

## Lease lifecycle

Each issued token is returned as a Vault *lease* of the `salesforce_access_token`
secret type. The lease TTL and max TTL come from the role's `ttl` and `max_ttl`.

- **Renew** extends the lease up to its max TTL while the cached token remains
  fresh.
- **Revoke** clears the role's cached token. Because the same Salesforce token is
  shared across every lease of a role, revoke does **not** call Salesforce's
  `/revoke` endpoint — doing so would invalidate the token for all other holders.
  The next read mints a new token.

To intentionally roll the shared token forward, use
[rotate](/vault/api-docs/secret/salesforce#rotate-a-role-token) instead.

## Concurrency

When several reads arrive on a cold or expired cache at the same time, a naive
engine would send one token request per read — a "mint stampede" that wastes
Salesforce's token allocation.

The engine prevents this with a per-role mint lock and double-checked caching:

1. **Fast path.** The read checks the cache without taking a lock. If the token
   is fresh, it returns immediately.
2. **Slow path.** On a miss, the read takes the role's mint lock. After acquiring
   the lock, it re-checks the cache, because another goroutine may have minted
   while it waited.
3. **Mint once.** Only the first reader through the lock calls Salesforce. The
   rest observe the freshly cached token.

This guarantees exactly one token request per role for a burst of concurrent
reads.
