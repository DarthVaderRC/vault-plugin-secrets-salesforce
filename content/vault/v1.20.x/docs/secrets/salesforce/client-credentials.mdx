# Tutorial: Client Credentials flow

This tutorial walks through issuing a Salesforce access token with the **Client
Credentials** grant. You configure a Connected App for the flow, create a Vault
configuration and role, read a token, call the Salesforce REST API, and manage
the token lifecycle.

The Client Credentials flow issues a token that acts as the Connected App's
"Run As" user. It is the simplest server-to-server flow because it needs only a
Consumer Key and Consumer Secret.

## Prerequisites

- The Salesforce secrets engine
  [enabled at `salesforce/`](/vault/docs/secrets/salesforce/setup).
- Permission to create or edit a Connected App in your Salesforce org.
- The `vault` and `curl` CLIs, and `jq` for readable output.

## Step 1: Configure the Connected App

In Salesforce, configure a Connected App for the Client Credentials flow:

1. Enable **My Domain** for the org if it is not already enabled.
2. In the Connected App's OAuth settings, enable **OAuth** and add the **Manage
   user data via APIs (api)** scope.
3. Enable the **Client Credentials Flow** and assign a **Run As** user. This user
   determines the token's permissions.
4. Set **Permitted Users** to "Admin approved users are pre-authorized" and assign
   the Run As user's Profile or a Permission Set to the app.
5. If the app enforces IP restrictions, relax them or allowlist Vault's egress IP.
6. Copy the **Consumer Key** and **Consumer Secret**.

## Step 2: Write the Vault configuration

Store the connection details and the Consumer Secret in Vault. The
`client_secret` is write-only and never returned.

```shell-session
$ vault write salesforce/config/prod-cc \
    login_url="https://acme.my.salesforce.com" \
    client_id="<ConsumerKey>" \
    client_secret="<ConsumerSecret>"
```

## Step 3: Create the role

Create a `client_credentials` role bound to the configuration:

```shell-session
$ vault write salesforce/roles/cc \
    config=prod-cc \
    grant_type=client_credentials \
    token_ttl=15m \
    ttl=10m \
    max_ttl=1h
```

Set `token_ttl` to match your org's session timeout so the cache does not serve a
token that Salesforce has already expired.

<Note>

The Client Credentials flow does not accept request scopes. The granted scopes
come from the Connected App's OAuth configuration, so you do not set `scopes` on
the role for this flow.

</Note>

## Step 4: Read a token

Read the credentials path to mint a token:

```shell-session
$ vault read salesforce/creds/cc
Key             Value
---             -----
lease_id        salesforce/creds/cc/AbCdEf...
lease_duration  10m
lease_renewable true
access_token    00D5g000004...
cached          false
expires_at      2026-06-25T06:15:00Z
grant_type      client_credentials
instance_url    https://acme.my.salesforce.com
token_type      Bearer
```

The first read shows `cached false`. Read again within the freshness window and
the engine serves the same token from the cache:

```shell-session
$ vault read -field=cached salesforce/creds/cc
true
```

## Step 5: Call the Salesforce REST API

Use the `access_token` against the returned `instance_url`:

```shell-session
$ TOKEN=$(vault read -field=access_token salesforce/creds/cc)
$ INSTANCE=$(vault read -field=instance_url salesforce/creds/cc)
$ curl -s -H "Authorization: Bearer $TOKEN" \
    "$INSTANCE/services/data/v60.0/limits" | jq .
```

A successful response confirms Vault minted a working token for the Run As user.

## Step 6: Manage the lifecycle

Each token read issues a Vault lease. Inspect or revoke it like any other lease:

```shell-session
$ LEASE=$(vault read -field=lease_id salesforce/creds/cc)
$ vault lease lookup "$LEASE"
$ vault lease revoke "$LEASE"
```

Revoking the lease clears the role's cached token; the next read mints a fresh
one. To force a new shared token without waiting for expiry, rotate the role:

```shell-session
$ vault write -f salesforce/roles/cc/rotate
Key         Value
---         -----
expires_at  2026-06-25T06:30:00Z
role        cc
rotated     true
```

## Troubleshooting

| Error | Likely cause |
|---|---|
| `invalid_client` | The Client Credentials flow is not enabled, no Run As user is assigned, or My Domain is not enabled. |
| `invalid_request: scope parameter not supported` | A `scopes` value was forwarded. Remove `scopes` from the role. |
| `invalid_grant` | The Run As user is not permitted on the app. Assign the user's Profile or a Permission Set. |
| Token host rejected | The `login_url` host is not a Salesforce domain. Correct it, or set `allow_non_salesforce_host=true` for a vetted gateway. |
