# Salesforce secrets engine

The Salesforce secrets engine brokers short-lived **Salesforce OAuth 2.0 access
tokens** through Vault. Vault holds the Salesforce Connected App secrets, mints
access tokens on demand using a server-to-server OAuth grant, caches one token
per role, and manages each token's lifecycle as a Vault lease.

Applications read a Vault path to receive a ready-to-use access token instead of
embedding a Consumer Secret or a JWT signing key in their own configuration.

<Note>

This is a community secrets engine, distributed as an external Vault plugin. It
is not built into Vault and is not supported by HashiCorp. Build and register the
plugin before you enable it. See
[Setup and configuration](/vault/docs/secrets/salesforce/setup).

</Note>

## Why use this engine

Salesforce does not ship a first-party Vault secrets engine for OAuth token
management. Without one, each application stores its own Salesforce credential
and mints tokens itself. This engine centralizes that work:

- **Secrets stay in Vault.** The Consumer Secret and the JWT private key are
  written to Vault once and are never returned by a read.
- **Consumers get short-lived tokens.** A single `read` returns a fresh access
  token and the instance URL to call.
- **Rotation and revocation are central.** Operators rotate or revoke tokens
  from Vault without redeploying applications.
- **Access is governed by Vault policy.** Standard Vault ACLs and audit logging
  apply to every token request.

## Supported flows

The engine supports the two Salesforce server-to-server OAuth grants. Neither
grant requires a user to interactively approve access, so both are suitable for
backend integrations.

| Grant flow | Identity | Secret material | Use when |
|---|---|---|---|
| **JWT Bearer** | A specific Salesforce user (`sub`) | RSA private key + uploaded certificate | You need tokens that act as a named integration user. |
| **Client Credentials** | The app's "Run As" user | Consumer Secret | You want the simplest app-level identity. |

For background on each grant, see [How it works](/vault/docs/secrets/salesforce/concepts).

## Features

- **Two server-to-server flows**: JWT Bearer (RS256 assertion) and Client
  Credentials.
- **Per-role token cache**: one token per role in the Vault barrier, re-minted
  near expiry so a burst of reads does not churn tokens.
- **Anti-stampede mint lock**: concurrent reads on a cold cache trigger exactly
  one request to Salesforce.
- **Lease lifecycle**: issued tokens are Vault leases that you can renew or
  revoke.
- **On-demand rotation**: force a fresh token with a single write.
- **Resilience**: transient failures (network, HTTP 429, HTTP 5xx) are retried
  with exponential backoff and jitter; a still-valid cached token is served if a
  re-mint briefly fails.
- **Security defaults**: secrets are write-only and redacted on read, the token
  endpoint host is restricted to Salesforce domains, and TLS verification is on
  by default.

## Architecture

The engine uses the same `config` / `roles` / `creds` model as Vault's database
and cloud secrets engines.

```text
config/<name>          roles/<name>                 creds/<name>  (read)
  connection +    <--   grant flow + identity   -->  issue / cache / lease a token
  secret material       + TTLs, bound to a            │
  (login_url,           single config                 ├── cache/<role> in the barrier
   client_id,                                         └── Vault lease (renew / revoke)
   client_secret,
   private_key)
```

- **`config/<name>`** holds the connection details and secret material for one
  Salesforce org and app.
- **`roles/<name>`** binds a config to a grant flow, an identity, scopes, and
  TTLs.
- **`creds/<name>`** (and the alias **`token/<name>`**) issues a leased access
  token for a role.
- **`roles/<name>/rotate`** discards the cached token and mints a fresh one.

For the full design: caching, lease mapping, and concurrency: see
[How it works](/vault/docs/secrets/salesforce/concepts).

## Get started

1. Set up a Salesforce Connected App and grant it a server-to-server OAuth flow.
2. [Enable and configure the engine](/vault/docs/secrets/salesforce/setup).
3. Follow the tutorial for your flow:
   - [Client Credentials flow](/vault/docs/secrets/salesforce/client-credentials)
   - [JWT Bearer flow](/vault/docs/secrets/salesforce/jwt-bearer)

Before you deploy, review the
