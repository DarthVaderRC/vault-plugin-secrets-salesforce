# Set up the Salesforce secrets engine

This page explains how to build, register, enable, and configure the Salesforce
secrets engine. After you finish, follow the tutorial for your grant flow:
[Client Credentials](/vault/docs/secrets/salesforce/client-credentials) or
[JWT Bearer](/vault/docs/secrets/salesforce/jwt-bearer).

## Requirements

- Vault 1.12 or later with access to the `plugin_directory`.
- A Salesforce org with a Connected App (or External Client App) configured for a
  server-to-server OAuth flow. See the engine's `SALESFORCE-SETUP` guide for the
  Connected App walkthrough.
- The Go toolchain to build the plugin binary, matching the Vault server's
  platform.

## Build and register the plugin

The engine is a community plugin, so you build a binary, place it in Vault's
`plugin_directory`, and register it before enabling it.

1. Build the plugin for the Vault server's platform. A Vault plugin is a native
   binary for one `GOOS`/`GOARCH`:

   ```shell-session
   $ make build         # host platform
   $ make build-linux   # linux/arm64
   ```

   The binary is written to `dist/vault-plugin-secrets-salesforce`.

2. Copy the binary into the configured `plugin_directory`, then register it with
   its SHA-256 sum:

   ```shell-session
   $ SHA=$(sha256sum vault-plugin-secrets-salesforce | cut -d' ' -f1)
   $ vault plugin register \
       -sha256="$SHA" \
       -command="vault-plugin-secrets-salesforce" \
       secret vault-plugin-secrets-salesforce
   ```

## Enable the engine

Enable the plugin at a mount path. The examples in this documentation use
`salesforce/`:

```shell-session
$ vault secrets enable -path=salesforce vault-plugin-secrets-salesforce
Success! Enabled the vault-plugin-secrets-salesforce secrets engine at: salesforce/
```

## Configure a connection

Write a configuration for your Salesforce org and app. Provide the secret
material for the flow you intend to use: `client_secret` for Client Credentials,
or `private_key` for JWT Bearer.

```shell-session
$ vault write salesforce/config/prod \
    login_url="https://acme.my.salesforce.com" \
    client_id="<ConsumerKey>" \
    client_secret="<ConsumerSecret>"
```

The `client_secret` and `private_key` fields are write-only. A read of the
configuration shows `<redacted>` for any secret that is set:

```shell-session
$ vault read salesforce/config/prod
Key                          Value
---                          -----
allow_non_salesforce_host    false
ca_cert                      n/a
client_id                    3MVG9...
client_secret                <redacted>
login_url                    https://acme.my.salesforce.com
private_key                  ""
tls_skip_verify              false
token_url                    https://acme.my.salesforce.com/services/oauth2/token
```

### Config field reference

| Field | Required | Description |
|---|---|---|
| `login_url` | yes | Base login host, for example `https://login.salesforce.com`, `https://test.salesforce.com`, or `https://<MyDomain>.my.salesforce.com`. |
| `client_id` | yes | Connected App / External Client App Consumer Key (OAuth `client_id`). |
| `client_secret` | for Client Credentials | Consumer Secret. Write-only; never returned. |
| `private_key` | for JWT Bearer | PEM RSA private key for assertion signing. Write-only; never returned. |
| `token_url` | no | Override the full token endpoint. Defaults to `<login_url>/services/oauth2/token`. |
| `ca_cert` | no | PEM CA bundle to validate the TLS endpoint. |
| `tls_skip_verify` | no | Disable TLS verification (sandbox/testing only). Default `false`. |
| `allow_non_salesforce_host` | no | Permit a non-Salesforce token host (for example, a private gateway). Default `false`. |

## Configure a role

Bind the configuration to a grant flow, an identity, and TTL policy.

For Client Credentials:

```shell-session
$ vault write salesforce/roles/sales-sync \
    config=prod \
    grant_type=client_credentials \
    token_ttl=15m \
    ttl=10m \
    max_ttl=1h
```

For JWT Bearer, also provide the run-as `username` and, for many orgs, an
`audience` override:

```shell-session
$ vault write salesforce/roles/sales-sync \
    config=prod \
    grant_type=jwt_bearer \
    username="integration@acme.com" \
    audience="https://login.salesforce.com" \
    token_ttl=15m \
    ttl=10m
```

The engine validates the role when you write it: a `jwt_bearer` role requires
`username` and a config `private_key`; a `client_credentials` role requires a
config `client_secret`.

### Role field reference

| Field | Required | Description |
|---|---|---|
| `config` | yes | The `config/<name>` this role uses. |
| `grant_type` | yes | `jwt_bearer` or `client_credentials`. |
| `username` | for JWT Bearer | Salesforce username for the JWT `sub` (run-as identity). |
| `scopes` | no | OAuth scopes. Client Credentials ignores request scopes; set them on the app. |
| `token_ttl` | no | Assumed token lifetime; drives cache expiry. Default `15m`. |
| `ttl` | no | Default Vault lease TTL for issued tokens. |
| `max_ttl` | no | Maximum Vault lease TTL for issued tokens. |
| `renew_skew` | no | Re-mint this long before expiry. Default `60s`. |
| `jwt_expiry` | no | JWT `exp` window. Default `3m`, maximum `5m`. |
| `audience` | no | Override the JWT `aud`. Defaults to the config `login_url`. |

## Verify

Read the credentials path to confirm the engine can mint a token:

```shell-session
$ vault read salesforce/creds/sales-sync
Key             Value
---             -----
lease_id        salesforce/creds/sales-sync/AbCdEf...
lease_duration  10m
lease_renewable true
access_token    00D5g000004...
cached          false
expires_at      2026-06-25T06:15:00Z
grant_type      client_credentials
instance_url    https://acme.my.salesforce.com
token_type      Bearer
```

A second read within the freshness window returns the same token with
`cached    true`.
