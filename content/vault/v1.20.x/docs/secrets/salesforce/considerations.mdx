# Considerations and limitations

Review these Salesforce platform behaviors and plugin constraints before you
deploy the engine. None of them block normal use, but several shape how you
configure roles and set up your Connected App.

## Salesforce platform limits

The engine's cache-and-share design keeps it well inside Salesforce's limits.
The points below summarize the analysis behind that conclusion.

### Token minting does not consume your API allocation

Salesforce's 24-hour API allocation counts **data** API calls (REST, SOAP, Bulk,
and most Connect REST APIs). OAuth authentication calls to the token endpoint are
not in that list and do not draw down the allocation. Caching still reduces auth
round-trips and latency, but it does not preserve your daily data-API budget.

### Concurrent request limits do not apply to token minting

Salesforce limits concurrent requests that run 20 seconds or longer (25 for
Production and Sandbox orgs, 5 for Developer and Trial orgs). Token mints are
sub-second, so this limit is effectively irrelevant to the engine. The per-role
mint lock further guarantees a single in-flight token request per role.

### Salesforce reuses tokens, so you do not accumulate them

Salesforce caps the number of outstanding OAuth tokens per user and Connected
App and evicts the oldest when the cap is exceeded. In practice this is not a
concern for the engine: real-org testing showed that repeated grants for the same
user, app, and scope return the **identical** access token while it remains valid,
because Salesforce de-duplicates server-side.

Combined with the engine's single-cached-token-per-role model, you do not
accumulate tokens toward any per-user or per-app cap under normal use.

<Warning>

Aggressive use of `rotate`, or any pattern that forces distinct tokens (for
example, explicit `/revoke` between mints), can defeat this de-duplication and
push you toward the eviction cap. Prefer the cache and rotate only when you need
to.

</Warning>

## Token lifetime is configured, not discovered

Salesforce token responses usually omit `expires_in`. The token lifetime is
governed by the Connected App session policy and the org session-timeout
settings, not by the response. The engine therefore treats lifetime as the role's
`token_ttl` parameter.

Set `token_ttl` to match the session timeout in your org. If `token_ttl` is
longer than the real session lifetime, the cache can serve a token that Salesforce
has already expired, causing intermittent `401` responses.

## Shared per-role token semantics

All consumers and all leases of a role share **one** Salesforce access token.
This is safe — Salesforce access tokens are stateless bearer tokens — but it
changes how revocation behaves:

- Revoking a Vault lease clears the cached token but does **not** call
  Salesforce's `/revoke`, because that would invalidate the token for every other
  holder.
- To roll the shared token forward for everyone, use
  [rotate](/vault/api-docs/secret/salesforce#rotate-a-role-token).

If you need per-consumer token isolation, create a separate role per consumer.

## Setup prerequisites in Salesforce

These are one-time Connected App setup requirements, surfaced here so token reads
do not fail at runtime. See
[Setup and configuration](/vault/docs/secrets/salesforce/setup) for the full
procedure.

- **JWT Bearer pre-authorization.** The run-as user must be pre-authorized for the
  Connected App — either through "Admin approved users are pre-authorized" plus a
  Permission Set or Profile assignment, or a prior user approval. Otherwise the
  token call fails with `invalid_grant: user hasn't approved this consumer`.
- **Client Credentials prerequisites.** The org needs My Domain, the flow must be
  enabled on the app, and a run-as user must be assigned. Missing any of these
  produces `invalid_client` or flow-not-enabled errors.
- **My Domain endpoints.** Salesforce steers orgs to
  `https://<MyDomain>.my.salesforce.com` OAuth endpoints. Set `login_url`
  accordingly; use `https://test.salesforce.com` or a sandbox My Domain for
  sandboxes.
- **IP restrictions.** If the Connected App enforces IP ranges, allowlist Vault's
  egress IP or relax IP enforcement on the app, or token requests are rejected.
- **Clock accuracy (JWT Bearer).** Salesforce rejects assertions whose `exp` is
  too far in the future. Keep `jwt_expiry` at or below 5 minutes (the engine
  enforces this) and ensure the Vault host clock is accurate.

## Plugin constraints

- **Single platform per binary.** A Vault plugin is a native binary for one
  GOOS/GOARCH. Build the binary for the platform of the Vault server that runs it.
  Build a separate binary for each platform in a mixed fleet.
- **Community plugin.** The engine is an external plugin, not built into Vault.
  Register it before you enable it.

## Out of scope

The engine intentionally does not implement:

- The refresh-token, web-server, device, or username-password OAuth flows.
- Managing Connected Apps inside Salesforce.
- Calling Salesforce's `/revoke` endpoint (by design, given the shared-token
  model).

The `use_introspection` role field is reserved for a future option that validates
token expiry via the Salesforce introspection endpoint. It currently has no
effect.
