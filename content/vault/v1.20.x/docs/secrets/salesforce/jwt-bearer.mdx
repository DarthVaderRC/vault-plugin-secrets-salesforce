# Tutorial: JWT Bearer flow

This tutorial walks through issuing a Salesforce access token with the **JWT
Bearer** grant. You create a signing keypair, configure a Connected App to trust
it, store the private key in Vault, create a role, read a token, call the
Salesforce REST API, and manage the token lifecycle.

The JWT Bearer flow issues a token that acts as a specific Salesforce user (the
role's `username`). Vault holds the RSA private key and signs an assertion;
Salesforce verifies it with the uploaded certificate. No Consumer Secret is
required.

## Prerequisites

- The Salesforce secrets engine
  [enabled at `salesforce/`](/vault/docs/secrets/salesforce/setup).
- Permission to create or edit a Connected App in your Salesforce org.
- The `vault`, `curl`, and `openssl` CLIs, and `jq` for readable output.

## Step 1: Create a signing keypair

Vault holds the private key; Salesforce holds the public certificate.

```shell-session
$ openssl req -x509 -nodes -newkey rsa:2048 \
    -keyout sf_jwt.key -out sf_jwt.crt \
    -subj "/CN=vault-salesforce-engine" -days 365
```

- `sf_jwt.key` is the private key for the Vault configuration.
- `sf_jwt.crt` is the certificate you upload to the Connected App.

Confirm the pair matches — the two digests must be identical:

```shell-session
$ openssl rsa  -in sf_jwt.key -noout -modulus | openssl md5
$ openssl x509 -in sf_jwt.crt -noout -modulus | openssl md5
```

## Step 2: Configure the Connected App

In Salesforce, configure a Connected App for the JWT Bearer flow:

1. In the OAuth settings, enable **OAuth** and add the **Manage user data via APIs
   (api)** and **Perform requests at any time (refresh_token, offline_access)**
   scopes.
2. Enable **Use digital signatures** and upload `sf_jwt.crt`.
3. Set **Permitted Users** to "Admin approved users are pre-authorized".
4. Pre-authorize the integration user by assigning their Profile or a Permission
   Set to the app. Without this, the token call fails with
   `invalid_grant: user hasn't approved this consumer`.
5. If the app enforces IP restrictions, relax them or allowlist Vault's egress IP.
6. Copy the **Consumer Key**.

## Step 3: Write the Vault configuration

Store the connection details and the private key in Vault. The `private_key` is
write-only and never returned.

```shell-session
$ vault write salesforce/config/prod-jwt \
    login_url="https://acme.my.salesforce.com" \
    client_id="<ConsumerKey>" \
    private_key=@sf_jwt.key
```

## Step 4: Create the role

Create a `jwt_bearer` role with the run-as `username`. Many orgs require the
`audience` to be `https://login.salesforce.com` even when `login_url` is a My
Domain host.

```shell-session
$ vault write salesforce/roles/jwt \
    config=prod-jwt \
    grant_type=jwt_bearer \
    username="integration@acme.com" \
    audience="https://login.salesforce.com" \
    token_ttl=15m \
    ttl=10m
```

The engine signs the assertion with the claims `iss` (client ID), `sub`
(username), `aud` (the audience), and `exp` (now plus `jwt_expiry`, default
3 minutes, capped at 5 minutes).

## Step 5: Read a token

Read the credentials path to mint a token:

```shell-session
$ vault read salesforce/creds/jwt
Key             Value
---             -----
lease_id        salesforce/creds/jwt/AbCdEf...
lease_duration  10m
lease_renewable true
access_token    00D5g000004...
cached          false
expires_at      2026-06-25T06:15:00Z
grant_type      jwt_bearer
instance_url    https://acme.my.salesforce.com
token_type      Bearer
```

A second read within the freshness window serves the same token from the cache:

```shell-session
$ vault read -field=cached salesforce/creds/jwt
true
```

## Step 6: Call the Salesforce REST API

Use the `access_token` against the returned `instance_url`:

```shell-session
$ TOKEN=$(vault read -field=access_token salesforce/creds/jwt)
$ INSTANCE=$(vault read -field=instance_url salesforce/creds/jwt)
$ curl -s -H "Authorization: Bearer $TOKEN" \
    "$INSTANCE/services/data/v60.0/limits" | jq .
```

The API call runs with the permissions of the role's `username`.

## Step 7: Manage the lifecycle

Each token read issues a Vault lease:

```shell-session
$ LEASE=$(vault read -field=lease_id salesforce/creds/jwt)
$ vault lease lookup "$LEASE"
$ vault lease revoke "$LEASE"
```

Revoking the lease clears the cached token; the next read mints a fresh one. To
force a new shared token immediately, rotate the role:

```shell-session
$ vault write -f salesforce/roles/jwt/rotate
Key         Value
---         -----
expires_at  2026-06-25T06:30:00Z
role        jwt
rotated     true
```

## Troubleshooting

| Error | Likely cause |
|---|---|
| `invalid_grant: user hasn't approved this consumer` | The run-as user is not pre-authorized. Assign their Profile or a Permission Set to the app. |
| `invalid_grant: invalid assertion` | The uploaded certificate does not match the private key, or the assertion `aud`/`sub` is wrong. |
| `invalid_grant` with a clock complaint | The Vault host clock is skewed, or `jwt_expiry` is too large. Keep it at or below 5 minutes. |
| Token host rejected | The `login_url` host is not a Salesforce domain. Correct it, or set `allow_non_salesforce_host=true` for a vetted gateway. |
