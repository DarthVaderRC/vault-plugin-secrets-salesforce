# Production hardening

The Salesforce secrets engine ships with resilience and security controls that
make it safe to run in production. This page describes those controls and the
operator practices that complement them.

## Resilience

### Anti-stampede mint lock

A burst of concurrent reads on a cold or expired cache produces exactly one token
request to Salesforce. The engine uses a per-role lock and double-checked caching
so that only the first reader mints, while the rest receive the freshly cached
token. See [Concurrency](/vault/docs/secrets/salesforce/concepts#concurrency) for
the full algorithm.

### Transient-failure retry

Token requests retry automatically when they hit a transient failure:

- Network and transport errors.
- HTTP 429 (rate limited).
- HTTP 5xx (server errors).

Retries use **exponential backoff with jitter** (base 200ms, up to 3 attempts)
and respect the request context, so a canceled or timed-out request stops
retrying immediately. Client errors (HTTP 4xx other than 429) fail fast, because
retrying a bad request or a rejected assertion will not succeed.

### Graceful degradation

If a re-mint fails but the previously cached token is still valid — that is, the
read happened inside the `renew_skew` window and Salesforce is briefly
unavailable — the engine logs a warning and serves the still-valid cached token
instead of failing the read. This keeps consumers working through short
Salesforce outages.

## Security

### Write-only, redacted secrets

The `client_secret` and `private_key` fields are write-only. The engine stores
them in the Vault barrier and never returns them. A configuration read shows
`<redacted>` for any secret that is set.

Because writing a configuration sets secret material, treat `config/*` write
access as equivalent to holding those secrets and restrict it to trusted admins.

### Token endpoint host allowlist

By default the engine refuses to send credentials to a token endpoint outside a
Salesforce domain. The allowed DNS suffixes are:

- `.salesforce.com`
- `.force.com`
- `.salesforce.mil` (Government Cloud)

This is defense in depth: it prevents a misconfiguration or a malicious config
write from exfiltrating the Consumer Secret or a JWT assertion to an attacker's
host. The engine validates the effective token endpoint when you write a
configuration, so a disallowed host fails fast.

The engine also requires `https` for the token endpoint (loopback addresses are
exempt so that test servers work).

<Warning>

Only set `allow_non_salesforce_host=true` for a vetted private gateway that you
control. It disables the host allowlist for that configuration.

</Warning>

### TLS verification

TLS verification is on by default. Provide a custom CA bundle with `ca_cert` when
the endpoint presents a private certificate. The `tls_skip_verify` option exists
for sandbox and testing only; never enable it in production.

### No secret leakage in errors

The engine maps Salesforce OAuth errors to Vault errors using only the `error`
and `error_description` fields from the response. Request secrets — the Consumer
Secret and the signed assertion — are never echoed back in an error message or
the Vault audit log.

## Least-privilege access

Grant Vault policies that match each consumer's role. The engine separates three
kinds of access:

- **Applications** read tokens for the specific role they own.
- **Operators** manage roles and force rotation without reading connection
  secrets.
- **Admins** manage connection configurations, which hold the secret material.

### Application — read one role's tokens

```hcl
# salesforce-app-read.hcl
path "salesforce/creds/sales-sync" {
  capabilities = ["read"]
}

# Optional alias path.
path "salesforce/token/sales-sync" {
  capabilities = ["read"]
}
```

Grant per role. Avoid wildcarding `salesforce/creds/*` unless a workload
legitimately needs every role.

### Operator — manage roles and rotate

```hcl
# salesforce-operator.hcl
path "salesforce/roles/*" {
  capabilities = ["create", "read", "update", "delete", "list"]
}

# Force a fresh token, for example during an incident.
path "salesforce/roles/+/rotate" {
  capabilities = ["update"]
}

# Read tokens for verification.
path "salesforce/creds/*" {
  capabilities = ["read"]
}
```

### Admin — manage connection secrets

```hcl
# salesforce-admin.hcl
path "salesforce/config/*" {
  capabilities = ["create", "read", "update", "delete", "list"]
}

path "salesforce/roles/*" {
  capabilities = ["create", "read", "update", "delete", "list"]
}
```

### Rotation automation — rotate only

```hcl
# salesforce-rotator.hcl
path "salesforce/roles/+/rotate" {
  capabilities = ["update"]
}
```

## Audit and observability

- Every token request, rotation, and revocation flows through the Vault audit
  log, with secrets HMAC'd by Vault's audit device.
- The engine emits structured, secret-free logs: token mints at debug level,
  rotations at info, revocations at debug, and graceful-degradation fallbacks at
  warn.

## Operational notes

- **Shared per-role token.** All leases of a role share one Salesforce token.
  Revoking a lease clears the cache but does not invalidate the token at
  Salesforce for other holders. Use
  [rotate](/vault/api-docs/secret/salesforce#rotate-a-role-token) to roll the
  shared token forward.
- **Match `token_ttl` to the org session timeout** so the cache never serves a
  token that Salesforce has already expired.
